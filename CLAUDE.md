Instructions for **Claude Code** working in this repository. This file is yours to maintain — keep it

edit other agents' instruction files (e.g. `.github/copilot-instructions.md`).

## Project

`@ngui/auto-complete` — an Angular autocomplete that works both as a **directive** (`[ngui-auto-complete]`,
attached to any input) and as a standalone **component** (`<ngui-auto-complete>`).

Angular CLI monorepo with two projects:

- `projects/auto-complete` — the published library (this is what ships to npm).
- `projects/demo` — the demo app, built to `docs/` and deployed to GitHub Pages by CI.

The demo **consumes the built library from `dist/`**, not the source — rebuild the library before the demo
(or your editor) reflects library changes. The library's major version tracks the supported Angular major
(`@ngui/auto-complete@N` ⇒ Angular `N`); don't introduce a major that doesn't line up with an Angular major.
When bumping the Angular major, also update the static **Angular badge** in `README.md` (it's hard-coded,
e.g. `Angular-22`).

## Commands (from the repo root)

This repo uses **pnpm** (run `corepack enable` once; pnpm ships with Node).

```bash
pnpm install
pnpm run build-lib:prod   # build the library to dist/ (run BEFORE build-docs / before testing the demo)
pnpm run build-docs       # production build of the demo app into docs/
pnpm run lint             # ng lint (ESLint flat config, eslint.config.js)
pnpm exec ng test auto-complete --watch=false   # Vitest + jsdom (@angular/build:unit-test, no browser)
pnpm exec ng test demo --watch=false
pnpm run format:check     # prettier --check (CI gate); pnpm run format to fix
pnpm run cypress:run      # e2e
```

## Pre-PR validation gate

Before opening any PR for a code change, all must pass:

1. `pnpm run build-lib:prod`
2. `pnpm run build-docs` (watch for `NG8113` unused-import warnings)
3. `pnpm run lint`
4. `pnpm exec ng test auto-complete --watch=false`
5. `pnpm exec ng test demo --watch=false`
6. `pnpm run format:check`

## How to work

- **Plan non-trivial work first.** For anything with several steps, lay out a short plan / todo list before
  editing, and keep it updated.
- **One concern per branch/PR.** Small, focused changes are easier to review and to changelog.
- **Fix root causes, not symptoms**, and keep the change minimal — touch only what the task needs.
- **Verify before claiming done** — run the validation gate; report failures honestly with their output.
- **Branch from `master`, open the PR, then stop.** Let the maintainer review and merge; don't self-merge.
- **Docs travel with the code:** update `CHANGELOG.md` for every user-facing change, `README.md` when the
  public API/usage changes, and add a `MIGRATION.md` section for breaking changes.

## Conventions

- **Conventional Commits**; breaking changes use `!` in the title and a `BREAKING CHANGE:` footer.
- **Prettier-enforced** (`.prettierrc.json`: printWidth 140, tabs, single quotes, trailing commas) via a
  husky + lint-staged pre-commit hook. Markdown is in `.prettierignore`.
- **`projects/demo/src/app/build-info.ts` is generated, not tracked:** it's git-ignored and regenerated by
  the `postinstall` / `prebuild-docs` / `prestart` hooks, so it never appears in `git status`. Don't add it
  back to git.
- Demo template diagnostics in the editor can be **stale until you rebuild the library** — trust
  `build-docs` over the IDE.

## Library specifics (gotchas)

- **Inputs are signal `input()`s** (read-only). In unit tests set them with
  `fixture.componentRef.setInput('<public-alias>', value)`, not `component.x = …`. The directive forwards
  inputs to the dynamically created dropdown component via `ComponentRef.setInput()` for the same reason.
- **Keep public inputs' kebab-case aliases** (`min-chars`, `list-formatter`, …) — renaming for aesthetics
  breaks every consumer template for no functional gain.
- **`output()` defaults to `<void>`** — use `output<T>()` for events that carry a payload.
- Numeric inputs use `numberAttribute`, booleans use `booleanAttribute`, so `min-chars="2"` and
  `[min-chars]="2"` are both accepted and correctly typed.
- **Angular runs zoneless by default (v21+).** If a test mutates a child component's fields out of band,
  call `markForCheck()` on that child's `ChangeDetectorRef` before `detectChanges()` to avoid a false
  `NG0100` (`ExpressionChangedAfterItHasBeenChecked`).
- On `ng update`, pin **all** `@angular/*` packages to the same major, and check the `package.json` diff for
  silently-narrowed `eslint`/`typescript-eslint` ranges.
- **Unit tests run on Vitest + jsdom** via `@angular/build:unit-test` (no Karma, no `test.ts`). The builder
  auto-initializes the Angular TestBed with Vitest `globals: true`, so specs use bare `describe/it/expect`
  and there's no `initTestEnvironment`/`platform-browser-dynamic`. `tsconfig.spec.json` sets
  `types: ["vitest/globals", "node"]`. **The base `tsconfig.json` must NOT set an explicit `typeRoots`** —
  the builder's angular-compiler resolves `types` entries strictly against `typeRoots` (unlike `tsc`, which
  falls back to module resolution), so a `node_modules/@types`-only `typeRoots` makes `vitest/globals`
  unresolvable (`TS2304: Cannot find name 'expect'`).
